= CM5 Programming Jig Image

A buildable `rpi-image-gen` configuration for the host controller of a
Raspberry Pi CM5 programming jig.

The image:

* Targets a Compute Module 5 (Lite variant, booting from NVMe) running Debian
  Trixie.
* Boots straight into the full Raspberry Pi Desktop (labwc/Wayland) and
  automatically logs in the operator account (named `jig` by default; can be
  renamed at first boot - see below).
* Auto-launches Chromium (maximized) at `http://localhost:3142`, the
  https://github.com/raspberrypi/rpi-sb-provisioner[rpi-sb-provisioner] web
  dashboard. The browser runs on top of the normal desktop, so the operator can
  minimize it to reach the panel and desktop.
* Installs the provisioning and jig-control stack: `rpi-sb-provisioner`,
  `cm5-programming-jig-gpio-config` (jig relay/RPIBOOT/status-LED control), and
  `rpi-systemd-gpio`.
* Installs `rpi-connect` (screen sharing + remote shell) and `magic-wormhole`
  (ad-hoc secure file transfer).

== Layout

[source,text]
----
contrib/src/cm5-programming-jig/
|-- config
|   `-- cm5-programming-jig.yaml   # main build config
|-- layer
|   `-- cm5-jig-desktop.yaml       # desktop + autologin + browser autostart layer
|-- pkgs
|   `-- rpi-preseed_0.1.0_all.deb  # vendored first-boot customisation tool
`-- README.adoc
----

First-boot customisation (operator password, hostname, SSH, Wi-Fi, Connect,
hardware interfaces) is handled by *rpi-preseed*, a small Raspberry Pi tool that
applies a single TOML file from the boot partition on first boot. The `.deb` is vendored under `pkgs/` and installed by name from the config's
`packages:` list.

== Building

From the `rpi-image-gen` root:

[source,sh]
----
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ -c cm5-programming-jig.yaml
----

`-S` puts this directory at the front of the config and layer search paths, so
both `cm5-programming-jig.yaml` and the `cm5-jig-desktop` layer are found
automatically.

== Operator account credentials

There is *no default OS password*. Nothing is committed in the config, and the
`jig` account is created with a locked password unless one is supplied. Desktop
autologin works regardless (it does not need a password), so a locked account
still boots straight to the provisioner UI. If you supply no credential at all,
the unit is a sealed appliance with no admin path - see "If you configure
nothing" below before rolling out units.

The recommended way to set a per-unit password is at *flash time* with an
`rpi-preseed.toml` on the boot partition (option 0). The account can also be
given a credential at *build time* (options 1-3); overrides there are given after
`--` using the full `IGconf_<section>_<key>` variable name (not a dotted
`section.key` form).

=== Option 0 (recommended): set the password per unit with rpi-preseed

The image installs *rpi-preseed*, which applies `/boot/firmware/rpi-preseed.toml`
on first boot, reusing the official Raspberry Pi helpers (`userconf`,
`imager_custom`, `raspi-config nonint`). Two ways to provide the file:

. *Raspberry Pi Imager* - use its OS customisation step in the flashing wizard
  (recent Imager versions present it in-line rather than as a separate "Edit
  Settings" dialog). Set a username (any name - defaults to `jig`) and a
  password. Imager writes `rpi-preseed.toml` for you.
. *By hand* - the boot (FAT) partition ships an inert, fully commented template
  `rpi-preseed.toml.example` plus `README-rpi-preseed.txt`. Mount the boot
  partition, copy `rpi-preseed.toml.example` -> `rpi-preseed.toml`, edit, and
  boot. The `.example` suffix keeps it inactive until renamed (rpi-preseed's
  units look for exactly `rpi-preseed.toml`).

On first boot the password is applied to the operator account, nothing is shared
between units, and no password is baked into the image. If you flash *without*
an `rpi-preseed.toml`, there is no seed: rpi-preseed does nothing and `jig` stays
locked (the desktop still autologins).

.How the account is configured (the operator name is free)
[NOTE]
====
rpi-preseed's `[user]` applier renames the UID 1000 account to `[user].name`,
moves its home directory, and sets its password via the RPi `userconf` helper
(including yescrypt hashing of a plaintext password on-device). The built-in
operator account is `jig`, but `[user].name` can be *any* name: a first-boot
unit (`cm5-jig-autologin-sync.service`) points desktop autologin at whoever owns
UID 1000, ordered after the rename and before the display manager starts, so
autologin follows the renamed account. Keep `name = "jig"` to retain the default
(the rename is then a no-op); to set only a password, set `password` alongside
whatever `name` you choose. rpi-preseed redacts secrets in the TOML in place
after a successful apply, and applies once (run-once via a stamp in
`/var/lib/rpi-preseed`; re-apply with `sudo rpi-preseed apply`).
====

.Minimal hand-written `rpi-preseed.toml`
[source,toml]
----
config_version = "1.0"

[system]
hostname = "cm5-jig"

[user]
name = "jig"                 # any name; "jig" keeps the default
password = "Ch4nge-me!"      # plaintext, hashed on-device (yescrypt)
password_encrypted = false

[ssh]
enabled = true
password_authentication = false
authorized_keys = ["ssh-ed25519 AAAA...replace... you@host"]
----

[NOTE]
====
Wi-Fi is supported too: rpi-preseed's `[wlan]` section is written as a
NetworkManager connection (this image uses NetworkManager as its sole network
manager - see _Networking_). Set it in Imager or in the `[wlan]` block of the
TOML.
====

=== Option 1: no local password, administer by SSH key / Connect

Leave the password unset and log in with an SSH key (and/or Raspberry Pi
Connect). There is then no password to leak or share:

[source,sh]
----
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ \
  -c cm5-programming-jig.yaml -- "IGconf_ssh_pubkey_user1=$(< ~/.ssh/id_ed25519.pub)"
----

=== Option 2: hashed password (keeps cleartext out of shell history/logs)

Generate a crypt hash and pass that instead of the plaintext. `device-user-admin`
applies it via `chpasswd -e`, so any valid hash works:

[source,sh]
----
read -rs OPPW                                   # prompt, not echoed, not in history
HASH=$(printf '%s' "$OPPW" | openssl passwd -6 -stdin); unset OPPW
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ \
  -c cm5-programming-jig.yaml -- IGconf_device_user1passhash="$HASH"
----

The bundled `bin/genpasswd <rootfs>` helper does the same, selecting the target's
`ENCRYPT_METHOD` (yescrypt on Trixie) automatically.

=== Option 3: plaintext password at build time

Simplest, but the cleartext appears in your shell history and process list. Must
satisfy the complexity policy (>=8 chars with an uppercase letter, a lowercase
letter, a digit and one of `@$!%*?&`):

[source,sh]
----
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ \
  -c cm5-programming-jig.yaml -- IGconf_device_user1pass='Ch4nge-me!'
----

Setting a password (options 2 or 3) automatically grants password-based `sudo`
to the `jig` account.

NOTE: A password set at build time (options 2-3) is baked into the image, so
every unit written from it shares that password. For per-unit credentials use
option 0 (Raspberry Pi Imager) or option 1 (per-unit SSH keys).

=== If you configure nothing: the jig is a sealed appliance

If a unit is flashed with *no* credential configured by any of the routes above -
no `rpi-preseed.toml`/Imager customisation, no build-time password or hash, no
baked SSH key, and Connect not enrolled - it still works as an appliance but has
*no login or administration path at all*. Concretely, with the defaults:

* The `jig` account is created locked (`adduser --disabled-password`), so there
  is no console or SSH *password* login.
* `root` is disabled (`usermod --pass='*' root`).
* `jig` is *not granted sudo*: `device.user1sudo` defaults to `none` (it only
  becomes `passwd` when a password is set), so no sudoers entry is written and
  `jig` is not added to the `sudo` group. The `sudo` binary itself stays present
  (a base package, `raspinfo`, depends on it), but `jig` cannot use it - so even
  the auto-logged-in desktop session cannot escalate.

This is by design: an un-provisioned unit is sealed rather than shipping a shared
default credential. The kiosk still functions - it autologins `jig` and opens the
provisioner dashboard - but you cannot get a privileged shell locally or remotely.

[IMPORTANT]
====
To have *any* administrative access you must supply a credential when the unit is
prepared: a per-flash password via Raspberry Pi Imager (option 0), a build-time
password/hash (options 2-3), a baked SSH key (option 1), or an enrolled Raspberry
Pi Connect session. Decide this before rolling out units.

Recovering a unit that was sealed by accident means re-flashing it, or mounting
its storage on another machine to add a credential (e.g. drop an
`rpi-preseed.toml` onto the boot partition and reboot - see "Operator account
credentials" option 0 and the template on the boot partition).
====

== Networking

The image standardises on *NetworkManager* as the single network manager, so the
desktop panel's network applet works and rpi-preseed's `[wlan]` Wi-Fi
customisation is applied on first boot.

This needs a deliberate choice because two stacks are otherwise present:

* The `trixie-minbase` suite sets up *systemd-networkd + iwd* (enabled, with
  per-interface DHCP `.network` files) - the lean server stack.
* The `rpd-*` desktop metapackages pull in *NetworkManager + netplan +
  wpa_supplicant*, and `NetworkManager.service` is also enabled.

Running both races over `eth0`/`wlan0` and splits Wi-Fi across two backends
(iwd vs wpa_supplicant). The `cm5-jig-desktop` layer therefore masks
`systemd-networkd(.socket, -wait-online)` and `iwd`, removes the suite's
`01-eth0.network`/`02-wlan0.network`, and leaves NetworkManager as the sole
manager. (The null `*.link` files are kept - they are processed by udev, not
networkd - so interfaces stay named `eth0`/`wlan0`.)

.How Wi-Fi is applied
[NOTE]
====
rpi-preseed's `[wlan]` applier writes an NM keyfile directly at
`/etc/NetworkManager/system-connections/preconfigured.nmconnection` (WPA2/WPA3/
open, hidden SSID, country). That only takes effect with NetworkManager as the
manager, which is why the layer disables systemd-networkd/iwd. The layer also
sets netplan's default renderer to NetworkManager
(`/etc/netplan/00-cm5-jig-renderer.yaml`) so that any netplan config that does
appear renders to NM rather than the disabled networkd backend.
====

Set Wi-Fi in Raspberry Pi Imager, or in the `[wlan]` block of
`rpi-preseed.toml`. Both end up as an NM connection.

== Filling a larger disk (root filesystem expansion)

The image ships with a small fixed root partition. Expansion to fill a larger
NVMe/SD is handled by a small, self-contained first-boot unit,
`cm5-jig-growfs.service` (installed by the `cm5-jig-desktop` layer):

* `growpart` (from `cloud-guest-utils`) extends the root partition to the end of
  the disk, then `resize2fs` grows the ext4 filesystem into it. ext4 supports
  *online* resize, so this runs against the mounted root without a reboot.
* The unit resolves the root disk/partition dynamically (`findmnt` +
  `/sys/class/block/.../partition`), so it works on NVMe, SD or USB.
* It is idempotent and run-once: a stamp at `/var/lib/cm5-jig/growfs-done` (and a
  `ConditionPathExists=!` on it) means it runs once and then no-ops.

[NOTE]
====
This runs *unconditionally* on first boot - it does *not* depend on any rpi-preseed/Imager seed, so even a sealed unit
expands to fill its disk. (`systemd-repart` is not used: it is GPT-only and this
image uses an MBR table.)
====

== Raspberry Pi Connect

Connect is enabled to start at boot (`connect.on: y`). To enrol the device
unattended, provide an auth key generated in your Raspberry Pi Connect account
settings:

[source,sh]
----
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ \
  -c cm5-programming-jig.yaml -- IGconf_connect_authkey='<auth-key-or-path>'
----

Otherwise, sign in interactively on first use via `rpi-connect signin`.

== Changing the dashboard URL

The browser target defaults to the local provisioner UI. Override it with:

[source,sh]
----
rpi-image-gen build -S ./contrib/src/cm5-programming-jig/ \
  -c cm5-programming-jig.yaml -- IGconf_jigdesktop_url='http://localhost:3142/'
----

== Notes

* `rpi-sb-provisioner`'s `rpi-provisioner-ui` service listens on
  `127.0.0.1:3142` (HTTP) by default. The browser launcher waits for this
  service to accept connections before opening, so it does not flash a
  connection error at boot.
* The desktop uses the modular `rpd-*` metapackages
  (`rpd-wayland-core` + `rpd-theme` + `rpd-preferences`), which replace the
  obsolete `raspberrypi-ui-mods` package on Trixie. The heavier `rpd-wayland-all`
  set is intentionally not used to keep the manufacturing image lean.
